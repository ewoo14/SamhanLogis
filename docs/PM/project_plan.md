## 1. 프로젝트 개요

### 1.1 배경
- (주)삼한공조시스템은 삼성 시스템에어컨 공식 파트너사
- 현재 이카운트 + 구글 스크립트(견적서/주문서)로 업무 수행 중
- 자체 물류 프로그램(회계, 그룹웨어, 대시보드)으로 행정 전환 목표

### 1.2 핵심 요구사항 요약
| 구분 | 내용 |
|------|------|
| 아키텍처 | MSA, 분산 Docker 서버, 분산 DB, Eureka 기반 게이트웨이/로드밸런싱 |
| 배포 형태 | 내부용: EXE(Electron), 외부 거래처용: 웹/앱 |
| 조직 구조 | 대표실 → 사무실(행정) / 물류창고(초월·상일·가상창고) / 거래처 |
| 권한 체계 | 마스터, 개발자, 매니저, 영업원, 회계원, 창고원, 재고원 (7단계 세분화) |

## 3. 핵심 기능 상세 설계

### 3.1 전표 관리 시스템

#### 전표 번호 체계
- 형식: `YYYY/MM/DD - {전표번호}` (내림차순 정렬)
- 모든 전표에 UUID 부여

#### 출고일 변경 로직
1. 출고일 변경 시 → 새 날짜에서 마지막 번호+1 부여
2. 기존 날짜의 전표번호 → 자동 폐기 처리
3. 원래 날짜로 복원 시 → UUID 기반 선택권 제공:
   - **(A)** 기존 번호 복원
   - **(B)** 해당 날짜 마지막 번호+1 새로 부여

#### 수정 이력 추적 (Google Docs 스타일)
- 최초 저장 후 수정 시 수정 카운트 증가 (1차, 2차, ... N차)
- 수정 전 데이터: 취소선 표시
- 수정 후 데이터: 수정자별 고유 색상으로 표시
- 우측에 수정자 정보 + 색상 범례 표시
- 수정 이력 전체 조회 및 특정 시점 복원 기능

#### 전표 종류
- 출고전표, 입고전표, 입금전표, 출금전표

### 3.2 재고 관리 (FIFO + UUID)

- 모든 물품에 고유 UUID 부여
- 입고 시 UUID 생성 + 타임스탬프 기록
- 출고 시 가장 오래된 UUID부터 자동 선택 (FIFO)
- 물품 이동 이력 전체 추적 가능
- 창고 구분:
  - **초월창고**: 자체 소유, 내부 물류 적재/재고관리/입출고 (창고장 중심 팀)
  - **상일창고**: 임대창고
  - **온라인창고 / 서초창고**: 삼성 직배건 처리 가상창고

### 3.3 배송 태그 시스템

| 태그 | 입/출고 | 자동 규칙 |
|------|--------|----------|
| 당일 | 출고 | - |
| 야적 | 출고 | 특이사항에 `{출고일}상차 {다음날}하차` 자동 태깅 |
| 지방 | 출고 | 특이사항에 `{출고일}상차 {다음날}하차` 자동 태깅 |
| 로젠택배 | 출고 | - |
| 경동택배 | 출고 | - |
| 경동화물 | 출고 | - |
| 회차 | **입고** | - |
| 일부회차 | 출고 | - |
| 반품 | **입고** | - |
| 차용 | **입고** | - |
| 대여 | 출고 | - |
| 반납 | 출고 | - |

### 3.4 권한 체계 (7단계)

| 역할 | 접근 범위 | 비고 |
|------|----------|------|
| 마스터 | 전체 시스템 | 권한 조정 유일 담당 |
| 개발자 | 마스터 필적 | 마스터 승인 필요 |
| 매니저 | 위임 권한 범위 | 마스터가 위임할 권한 설정 |
| 영업원 | 견적서, 발주, 출고전표 | - |
| 회계원 | 입출금, 세금계산서, 원장 | - |
| 창고원 | 물류 입출고 처리 | - |
| 재고원 | 창고원 + 재고 심화 | 더 넓은 재고 권한 |

- 각 역할 내에서도 세부 접근 범위 분할 가능
- 권한 조정은 **오로지 마스터만** 가능

### 3.5 품목 관리
- 품목별 고유 UUID
- 필드: 품목명, 모델명, 출고가, 납품가
- 상세 스펙: 태그 형식 추가, 견적서에서 조회 가능

### 3.6 회계 시스템
- 법인계좌 입출금 내역/잔액 조회
- 세금계산서 일괄/개별 발행 (출고전표 단위)
- 거래처별 원장 관리
- 권한별 데이터 표시 차등

### 3.7 전체 감사 로깅
- **모든 작업**에 대해 감사 로그 기록
- 로그 항목: 작업자, 작업 내용, 실행 위치, 타임스탬프, IP, 변경 전/후 데이터

## 7. 개발 로드맵 (6단계, 총 28주)

### Phase 1: 기반 인프라 (4주)
- Docker 환경 구성, Eureka Server, API Gateway
- PostgreSQL 클러스터, Redis, RabbitMQ 설정
- 인증/권한 서비스 (JWT, 7단계 역할 기반 접근 제어)
- 로깅/감사 서비스 기본 구축
- 디자인 시스템 확립

### Phase 2: 핵심 도메인 (6주)
- 사용자/조직 서비스 (조직도, 역할 관리)
- 품목 서비스 (UUID, 태그, 스펙)
- 재고/창고 서비스 (FIFO, UUID 추적)
- Desktop(Electron) 앱 기본 프레임

### Phase 3: 전표 시스템 (6주)
- 전표 서비스 (출고·입고·입금·출금 전표)
- 전표 번호 체계 + 출고일 변경 로직
- 수정 이력 추적 (Google Docs 스타일)
- 배송 태그 시스템

### Phase 4: 회계/거래처 (4주)
- 회계 서비스 (입출금, 잔액, 원장)
- 거래처 서비스 + 거래처별 원장
- 세금계산서 발행 (일괄/개별)
- 권한별 데이터 표시 차등

### Phase 5: 그룹웨어/대시보드 (4주)
- 그룹웨어 (메신저, 일정, 전자결재)
- 대시보드 (통계, 리포트, 시각화)
- 알림 서비스 (WebSocket, 푸시)

### Phase 6: 외부 배포 + 최적화 (4주)
- 거래처 주문 웹 포털
- 모바일 앱 (React Native)

- 운영 배포 + 모니터링

## 9. GitHub 활용 계획

### 9.1 브랜치 전략
- `main`: 운영 배포 브랜치
- `develop`: 개발 통합 브랜치
- `feature/{서비스명}/{기능명}`: 기능 개발 브랜치
- `hotfix/{이슈번호}`: 긴급 수정 브랜치

### 9.2 코드리뷰 프로세스
1. 개발자가 `feature` 브랜치에서 작업 → PR 생성
2. TM이 코드리뷰 → 승인/수정 요청
3. PM이 최종 확인 → `develop` 머지
4. 단계 완료 시 → `develop` → `main` 머지 (대표 승인)

### 9.3 Issue & Discussion 활용
- **Issues**: 버그 리포트, 기능 요청, 작업 트래킹
- **Discussions**: 설계 토론, 리뷰 승인, 의사결정 기록
- **Projects**: 칸반 보드로 진행 현황 관리
