## 1. 프로젝트 개요

### 1.1 배경
- (주)삼한공조시스템은 삼성 시스템에어컨 공식 파트너사
- 현재 이카운트 + 구글 스크립트(견적서/주문서)로 업무 수행 중
- 자체 물류 프로그램(회계, 그룹웨어, 대시보드)으로 행정 전환 목표

### 1.2 핵심 요구사항 요약
| 구분 | 내용 |
|------|------|
| 아키텍처 | MSA, 분산 Docker 서버, **서비스별 독립 DB**, Eureka 기반 게이트웨이/로드밸런싱 |
| 배포 형태 | 내부용: EXE(Electron), 외부 거래처용: 웹/앱 |
| 조직 구조 | 대표실 → 사무실(행정) / 물류창고(초월·상일·가상창고) / 거래처 |
| 권한 체계 | 마스터, 개발자, 매니저, 영업원, 회계원, 창고원, 재고원 (7단계 세분화) |

## 3. 핵심 기능 상세 설계

### 3.1 전표 관리 시스템

#### 전표 번호 체계
- 형식: `YYYY/MM/DD - {전표번호}` (내림차순 정렬)
- 모든 전표에 UUID 부여

#### 출고일 변경 로직
1. 출고일 변경 시 → 새 날짜에서 마지막 번호+1 부여
2. 기존 날짜의 전표번호 → 자동 폐기 처리
3. 원래 날짜로 복원 시 → UUID 기반 선택권 제공:
   - **(A)** 기존 번호 복원
   - **(B)** 해당 날짜 마지막 번호+1 새로 부여

#### 수정 이력 추적 (Google Docs 스타일)
- 최초 저장 후 수정 시 수정 카운트 증가 (1차, 2차, ... N차)
- 수정 전 데이터: 취소선 표시
- 수정 후 데이터: 수정자별 고유 색상으로 표시
- 우측에 수정자 정보 + 색상 범례 표시
- 수정 이력 전체 조회 및 특정 시점 복원 기능

#### 전표 종류
- 출고전표, 입고전표, 입금전표, 출금전표, **이동전표**

#### 전표 상태 라이프사이클

전표는 아래 상태를 순서대로 거치며, 각 상태에 따라 수정 가능 여부가 결정됨:

```
[작성중] → [저장완료] → [전송완료] → [수락] → [처리중] → [처리완료]
   │          │            │                        │
   │          │            └── 반려 → [작성중]       └── [확정]
   │          │
   │          └── 수정 가능 (수정 이력 카운트 증가)
   │
   └── 자유 수정 (이력 미기록)
```

| 상태 | 수정 가능 | 설명 |
|------|----------|------|
| 작성중 | ✅ 자유 수정 | 최초 작성, 수정 이력 미기록 |
| 저장완료 | ✅ 이력 기록 수정 | 저장 후 수정 시 수정 카운트 증가 |
| 전송완료 | ⚠️ 긴급 수정만 | 창고로 전송 완료, 일반 수정 불가 |
| 수락 | 🔒 **수정 불가** | 창고 직원이 전표 수락, **데이터 락 발동** |
| 처리중 | 🔒 **수정 불가** | 출고 작업 진행 중 |
| 처리완료 | 🔒 **수정 불가** | 출고 완료 |
| 확정 | 🔒 **수정 불가** | 최종 확정 (세금계산서 연계 가능) |

#### 데이터 락킹 메커니즘

**목표**: 창고 직원이 전표를 수락하고 출고 처리 시 해당 전표의 수정을 즉시 차단

##### 락킹 전략 (2단계)

**1단계: 낙관적 락킹 (Optimistic Locking)**
- 전표 엔티티에 `version` 필드 적용
- 동시 수정 시도 시 버전 충돌 감지 → 나중 수정 거부
- 작성중/저장완료 상태에서 적용

**2단계: 비관적 락킹 (Pessimistic Locking)**
- 전표가 `수락` 이상 상태로 전환되면 DB 레벨 Row Lock 적용
- 수정 API 호출 시 상태 체크 → 즉시 `403 Forbidden` 반환
- Redis 분산 락으로 다중 서버 환경에서도 동기화 보장

```
[영업원 수정 시도]
    │
    ▼
상태 확인 ──→ 작성중/저장완료? ──→ 수정 허용 (낙관적 락)
    │
    └──→ 전송완료 이상? ──→ 🔒 수정 차단
                              │
                              └──→ "긴급 수정 요청" 버튼만 표시
```

##### 실시간 동기화 (WebSocket + SSE)
- 전표 상태 변경 시 **WebSocket으로 즉시 브로드캐스트**
- 영업원 화면: 전표가 `수락`됨 → 실시간으로 수정 버튼 비활성화
- 창고원 화면: 새 전표 전송 → 실시간 푸시 알림 + 목록 자동 갱신
- 네트워크 지연 대비 **Polling 폴백** (3초 간격)

#### 긴급 수정 요청 워크플로우

전표가 `수락` 이후 상태일 때 영업원이 수정이 필요한 경우:

```
[영업원]                    [시스템]                  [창고원/매니저]
   │                          │                          │
   ├── 긴급 수정 요청 ───────→│                          │
   │   (수정 사유 + 변경내용)  │                          │
   │                          ├── 알림 전송 ────────────→│
   │                          │   (푸시 + WebSocket)      │
   │                          │                          │
   │                          │←── 승인/반려 ────────────┤
   │                          │                          │
   │   ┌──────────────────────┤                          │
   │   │                      │                          │
   │   ▼                      │                          │
   │ [승인 시]                 │                          │
   │  · 전표 상태 → 일시해제   │                          │
   │  · 30분 타임아웃 설정     │                          │
   │  · 수정 완료 → 재락킹    │                          │
   │                          │                          │
   │ [반려 시]                 │                          │
   │  · 반려 사유 전달         │                          │
   │  · 전표 상태 유지 (락 유지)│                         │
```

##### 긴급 수정 요청 규칙
1. **수정 요청 시 필수 입력**: 수정 사유, 변경할 항목 + 변경 내용 미리보기
2. **승인 권한**: 해당 전표를 수락한 창고원 또는 매니저 이상
3. **타임아웃**: 승인 후 **30분** 내 수정 미완료 시 자동 재락킹
4. **이력 기록**: 긴급 수정 요청/승인/반려 전체 감사 로그 기록
5. **처리완료/확정 상태**: 긴급 수정 불가, 별도 **정정 전표** 발행으로 대체
6. **실시간 알림**: 요청 즉시 창고원 모바일/데스크톱에 푸시


### 3.2 재고 관리 (FIFO + UUID)

- 모든 물품에 고유 UUID 부여
- 입고 시 UUID 생성 + 타임스탬프 기록
- 출고 시 가장 오래된 UUID부터 자동 선택 (FIFO)
- 물품 이동 이력 전체 추적 가능
- 창고 구분:
  - **초월창고**: 자체 소유, 내부 물류 적재/재고관리/입출고 (창고장 중심 팀)
  - **상일창고**: 임대창고
  - **온라인창고 / 서초창고**: 삼성 직배건 처리 가상창고

### 3.4 권한 체계 (7단계)

| 역할 | 접근 범위 | 비고 |
|------|----------|------|
| 마스터 | 전체 시스템 | 권한 조정 유일 담당 |
| 개발자 | 마스터 필적 | 마스터 승인 필요 |
| 매니저 | 위임 권한 범위 | 마스터가 위임할 권한 설정 |
| 영업원 | 견적서, 발주, 출고전표 | - |
| 회계원 | 입출금, 세금계산서, 원장 | - |
| 창고원 | 물류 입출고 처리 | - |
| 재고원 | 창고원 + 재고 심화 | 더 넓은 재고 권한 |

- 각 역할 내에서도 세부 접근 범위 분할 가능
- 권한 조정은 **오로지 마스터만** 가능

### 3.5 품목 관리
- 품목별 고유 UUID
- 필드: 품목명, 모델명, 출고가, 납품가
- 상세 스펙: 태그 형식 추가, 견적서에서 조회 가능

### 3.6 회계 시스템
- 법인계좌 입출금 내역/잔액 조회 (**오픈뱅킹 API 직접 연동**)
- 세금계산서 일괄/개별 발행 (**홈택스 API 직접 연동**)
- 거래처별 원장 관리
- 권한별 데이터 표시 차등

### 3.8 거래처 주문 시스템
- **링크 공유 방식**: 공식 홈페이지 도메인 사용 (예: `order.samhan.co.kr/{token}`)
- 거래처별 고유 주문 링크 생성 → 링크 접속 시 주문 페이지 표시
- 로그인 불필요, 토큰 기반 인증으로 간편 접근
- 주문 완료 시 내부 시스템에 자동 반영 → 출고전표 생성 연계

### 3.9 모바일 앱 (듀얼 구성)

#### 거래처용 앱
- 주문 링크 접속 → 주문서 작성/조회
- 배송 현황 추적
- 주문 이력 확인

#### 내부 직원용 앱
- **권한별 메뉴 차등 표시** (역할에 따라 접속 가능 메뉴가 다름)
- 창고원: 출고전표 수신 → 확인 → 출고 처리 (핵심 워크플로우)
- 영업원: 견적/주문 현황 조회, 긴급 전표 확인
- 매니저: 승인 대기 항목 처리, 현황 대시보드
- 알림 푸시 (전표 도착, 승인 요청 등)

### 3.10 이카운트 데이터 마이그레이션

#### 마이그레이션 대상
- 품목 마스터 데이터
- 거래처 정보
- 입출고 이력
- 재고 현황
- 회계 데이터 (입출금, 세금계산서)

#### 음수 수량 변환 규칙
이카운트에서는 음수 수량으로 반대 방향 처리를 하지만, SamhanLogis에서는 명시적 전표 분리:

| 이카운트 원본 | 변환 후 (SamhanLogis) |
|-------------|---------------------|
| 출고전표 수량 < 0 (마이너스 출고) | → **입고전표** 생성 (양수 수량) |
| 입고전표 수량 < 0 (마이너스 입고) | → **출고전표** 생성 (양수 수량, 삼성 착하판정 회수 등) |

#### 변환 프로세스
1. 이카운트 데이터 CSV/API 추출
2. 데이터 정합성 검증 (누락, 불일치 확인)
3. 음수 수량 레코드 분류 및 전표 유형 변환
4. UUID 신규 발행 + 기존 데이터 매핑 테이블 생성
5. 스테이징 DB 적재 → 검증 → 프로덕션 반영
6. 변환 리포트 생성 (원본 vs 변환 결과 대조표)

### 3.11 창고 간 재고 이동 (Stock Transfer)

#### 개요
창고 간 재고를 이동할 때 **이동전표**를 발행하여 출발 창고에서 출고, 도착 창고에서 입고를 하나의 트랜잭션으로 처리한다.

#### 이동 가능 경로
모든 창고 간 양방향 이동 가능:

| # | 출발 창고 | 도착 창고 | 비고 |
|---|----------|----------|------|
| 1 | 초월창고 | 상일창고 | 실물 이동 (차량 배송) |
| 2 | 상일창고 | 초월창고 | 실물 이동 (차량 배송) |
| 3 | 초월창고 | 온라인창고/서초창고 | 가상 이동 (삼성 직배건 전환) |
| 4 | 상일창고 | 온라인창고/서초창고 | 가상 이동 (삼성 직배건 전환) |
| 5 | 온라인창고/서초창고 | 초월창고 | 가상 → 실물 전환 |
| 6 | 온라인창고/서초창고 | 상일창고 | 가상 → 실물 전환 |

#### 이동 사유 체계
이동전표 작성 시 **이동 사유** 필수 선택:

| 사유 코드 | 사유명 | 설명 |
|----------|--------|------|
| `REBALANCE` | 재고 재배치 | 창고 간 재고 균형 조정 |
| `URGENT` | 긴급 출고 대응 | 특정 창고에서 긴급 출고 필요 시 타 창고에서 이동 |
| `CONSOLIDATE` | 재고 통합 | 분산된 재고를 하나의 창고로 집중 |
| `MAINTENANCE` | 창고 정비 | 창고 정비/점검으로 인한 임시 이동 |
| `SAMSUNG_DIRECT` | 삼성 직배건 전환 | 삼성 직배건 처리를 위한 가상창고 이동 |
| `OTHER` | 기타 | 직접 입력 (사유 상세 필수) |

#### 이동전표 상태 라이프사이클

```
[이동요청] → [승인대기] → [승인완료] → [출고처리] → [운송중] → [입고완료] → [이동확정]
    │           │                                                    │
    │           └── 반려 → [이동요청] (수정 후 재요청)                  │
    │                                                                │
    └── 취소 가능 (승인 전)                                     최종 확정 (수정 불가)
```

| 상태 | 설명 | 수정 가능 |
|------|------|----------|
| 이동요청 | 이동전표 작성, 사유 입력 | ✅ 자유 수정 |
| 승인대기 | 창고장(출발 창고) 승인 대기 | ⚠️ 요청자만 취소 가능 |
| 승인완료 | 출발 창고 창고장 승인 | 🔒 수정 불가 |
| 출고처리 | 출발 창고에서 물품 출고 완료 | 🔒 수정 불가, 출발 창고 재고 차감 |
| 운송중 | 실물 이동 중 (가상창고 이동 시 즉시 스킵) | 🔒 수정 불가 |
| 입고완료 | 도착 창고에서 물품 입고 확인 | 🔒 수정 불가, 도착 창고 재고 증가 |
| 이동확정 | 최종 확정, 감사 로그 기록 | 🔒 수정 불가 |

#### 승인 프로세스

```
[요청자]                  [출발 창고장]              [도착 창고장]           [시스템]
   │                          │                        │                    │
   ├── 이동전표 작성 ─────────→│                        │                    │
   │   (출발/도착 창고,         │                        │                    │
   │    품목, 수량, 사유)       │                        │                    │
   │                          │                        │                    │
   │                          ├── 재고 확인             │                    │
   │                          ├── 승인/반려 ─────────────────────────────────→│
   │                          │                        │                    │
   │                          │   [승인 시]             │                    │
   │                          │                        │                    ├── 알림 전송
   │                          ├── 출고 처리             │                    │   (도착 창고장에게)
   │                          │                        │                    │
   │                          │                        ├── 입고 확인         │
   │                          │                        ├── 입고 완료 ───────→│
   │                          │                        │                    ├── 이동 확정
   │                          │                        │                    ├── 재고 반영
   │                          │                        │                    └── 감사 로그
```

##### 승인 규칙
1. **승인 권한**: 출발 창고의 **창고장** 또는 **매니저 이상**
2. **재고 검증**: 승인 시 출발 창고의 가용 재고 자동 확인 (부족 시 승인 차단)
3. **도착 확인**: 도착 창고 담당자가 실물 확인 후 입고 처리
4. **가상창고 이동**: 온라인창고/서초창고 이동 시 운송 단계 자동 스킵, 즉시 재고 반영
5. **FIFO 유지**: 이동 시 UUID 및 입고 타임스탬프 보존 → 도착 창고에서도 FIFO 순서 유지
6. **전체 감사 로그**: 이동 요청/승인/반려/출고/입고 전 과정 기록

#### 재고 반영 규칙

| 시점 | 출발 창고 | 도착 창고 | 비고 |
|------|----------|----------|------|
| 이동요청 | 변동 없음 | 변동 없음 | 이동 예정 수량으로 표시만 |
| 승인완료 | 가용 재고에서 **예약 차감** | 변동 없음 | 다른 출고에 해당 수량 배정 방지 |
| 출고처리 | **실제 재고 차감** | 변동 없음 | 물품이 출발 창고를 떠남 |
| 입고완료 | - | **실제 재고 증가** | 물품이 도착 창고에 도착 |
| 이동확정 | 확정 | 확정 | 최종 정산 |

### 3.7 전체 감사 로깅

## 7. 데이터 표준: 감사 필드 (BaseEntity)

> ⚠️ **모든 마이크로서비스의 모든 엔티티는 아래 7개 감사 필드를 필수로 포함해야 한다.**

### 7.1 BaseEntity 구조

```java
@MappedSuperclass
@EntityListeners(AuditingEntityListener.class)
public abstract class BaseEntity {

    @CreatedDate
    @Column(nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @CreatedBy
    @Column(nullable = false, updatable = false)
    private String createdBy;

    @LastModifiedDate
    private LocalDateTime modifiedAt;

    @LastModifiedBy
    private String modifiedBy;

    private LocalDateTime deletedAt;

    private String deletedBy;

    @Column(nullable = false)
    private Boolean isDeleted = false;
}
```

### 7.2 적용 규칙

| # | 규칙 | 설명 |
|---|------|------|
| 1 | **BaseEntity 상속 필수** | 모든 엔티티는 `extends BaseEntity` |
| 2 | **물리 삭제 금지** | `DELETE` 쿼리 금지, Soft Delete만 허용 |
| 3 | **Soft Delete** | `isDeleted = true` + `deletedAt` + `deletedBy` 설정 |
| 4 | **자동 필터링** | API 조회 시 `isDeleted = false` 조건 자동 적용 (`@Where` 또는 글로벌 필터) |
| 5 | **사용자 자동 주입** | `createdBy` / `modifiedBy` / `deletedBy`는 JWT 토큰의 사용자 ID 자동 주입 |
| 6 | **타임스탬프 자동** | `createdAt` / `modifiedAt`은 JPA Auditing으로 자동 설정 |

### 7.3 DB 컬럼 매핑

| 필드 | 컬럼명 | 타입 | NOT NULL | 기본값 |
|------|--------|------|----------|--------|
| createdAt | `created_at` | `TIMESTAMP` | ✅ | - |
| createdBy | `created_by` | `VARCHAR(50)` | ✅ | - |
| modifiedAt | `modified_at` | `TIMESTAMP` | ❌ | NULL |
| modifiedBy | `modified_by` | `VARCHAR(50)` | ❌ | NULL |
| deletedAt | `deleted_at` | `TIMESTAMP` | ❌ | NULL |
| deletedBy | `deleted_by` | `VARCHAR(50)` | ❌ | NULL |
| isDeleted | `is_deleted` | `BOOLEAN` | ✅ | `false` |

## 8. QA 교차 검증 프로세스

상세 프로세스: [`docs/PM/qa_cross_validation_process.md`](qa_cross_validation_process.md)

### 8.1 핵심 규칙

1. 각 팀 QA는 자기 팀 내부 QA 완료 후, **교차 QA 대상 팀**의 기능도 검증
2. 교차 QA 보고서에 **스크린샷 1장 이상 필수 첨부**
3. 교차 QA **FAIL** 시 해당 기능은 `develop` 머지 차단
4. 교차 QA 요청 접수 후 **48시간 이내** 보고서 제출

### 8.2 교차 검증 시점

| 시점 | 범위 |
|------|------|
| 기능 단위 | 각 기능 개발 완료 후, 연관 서비스 QA가 검증 |
| Phase 단위 | Phase 완료 전, 해당 Phase 모든 팀 QA 통합 검증 |
| 릴리스 전 | main 머지 전 전체 서비스 통합 검증 |

## 9. 개발 로드맵 (7단계, 총 33주)

### Phase 1: 기반 인프라 (4주)
- Docker 환경 구성, Eureka Server, API Gateway
- PostgreSQL 클러스터, Redis, RabbitMQ 설정
- 인증/권한 서비스 (JWT, 7단계 역할 기반 접근 제어)
- 로깅/감사 서비스 기본 구축
- 디자인 시스템 확립

### Phase 2: 핵심 도메인 (6주)
- 사용자/조직 서비스 (조직도, 역할 관리)
- 품목 서비스 (UUID, 태그, 스펙)
- 재고/창고 서비스 (FIFO, UUID 추적, **창고 간 재고 이동**)
- Desktop(Electron) 앱 기본 프레임

### Phase 3: 전표 시스템 (6주)
- 전표 서비스 (출고·입고·입금·출금·**이동** 전표)
- 전표 번호 체계 + 출고일 변경 로직
- 수정 이력 추적 (Google Docs 스타일)
- 배송 태그 시스템

### Phase 4: 회계/거래처 (4주)
- 회계 서비스 (입출금, 잔액, 원장)
- 거래처 서비스 + 거래처별 원장
- 세금계산서 발행 (일괄/개별)
- 권한별 데이터 표시 차등

### Phase 5: 그룹웨어/대시보드 (4주)
- 그룹웨어 (메신저, 일정, 전자결재)
- 대시보드 (통계, 리포트, 시각화)
- 알림 서비스 (WebSocket, 푸시)

### Phase 6: 외부 배포 + 모바일 (5주)
- 거래처 주문 링크 시스템 (공식 도메인 기반)
- 모바일 앱 듀얼 구성 (거래처용 + 내부 직원용)
- 내부 앱 권한별 메뉴 차등 구현
- 창고원 모바일 출고전표 수신/처리 워크플로우

### Phase 7: 이카운트 마이그레이션 + 최적화 (4주)
- 이카운트 데이터 추출 및 정합성 검증
- 음수 수량 변환 처리 (출고↔입고 전환)
- UUID 매핑 및 스테이징 검증
- 전체 통합 테스트, 성능 최적화
- 운영 배포 + 모니터링

#### 브랜치 규칙
| 브랜치 | 관리자 | 머지 대상 | 보호 설정 |
|--------|--------|----------|----------|
| `main` | 대표 승인 | `develop` → `main` | 강제 푸시 금지, PR 필수 |
| `develop` | PM | 팀 브랜치 → `develop` | PM 승인 필수, CI 통과 필수, **교차 QA PASS 필수** |
| `team/*` | 각 TL | 개인 브랜치 → `team/*` | TL 승인 필수 |
| 개인 브랜치 | 개발자 | - | 자유 푸시 |

### 11.2 머지 요청 프로세스 (TL → PM)

```
[개발자]              [TL (팀장)]              [PM]                [대표]
   │                     │                      │                    │
   ├── 개인 브랜치 작업   │                      │                    │
   ├── PR 생성 ────────→│                      │                    │
   │                     ├── 코드리뷰            │                    │
   │                     ├── 승인/수정요청        │                    │
   │                     │                      │                    │
   │                     ├── team/* 머지 완료    │                    │
   │                     ├── develop 머지 요청 ─→│                    │
   │                     │   (PR + 변경사항 요약) │                    │
   │                     │                      ├── 충돌 검토         │
   │                     │                      ├── 타 팀 영향 분석   │
   │                     │                      ├── 승인 / 반려       │
   │                     │                      │                    │
   │                     │                      │ [Phase 완료 시]     │
   │                     │                      ├── main 머지 요청 ─→│
   │                     │                      │                    ├── 최종 승인
```

#### PM 머지 검토 체크리스트

PM은 `develop` 머지 요청 시 아래 항목을 **모두 확인** 후 승인/반려 결정:

| # | 검토 항목 | 설명 |
|---|----------|------|
| 1 | **타 팀 충돌 여부** | 다른 팀 브랜치와 코드 충돌이 없는가? |
| 2 | **API 호환성** | 변경된 API가 다른 서비스의 기존 호출에 영향을 주지 않는가? |
| 3 | **DB 스키마 영향** | 독립 DB 간 데이터 정합성에 문제가 없는가? |
| 4 | **테스트 통과** | CI 파이프라인 (빌드 + 테스트)이 모두 통과했는가? |
| 5 | **문서 업데이트** | 관련 기획서/API 문서/함수 문서가 함께 업데이트되었는가? |
| 6 | **기획서 준수** | 기획서에 명시된 스펙과 일치하는가? |
| 7 | **필수 문서 제출** | 기획서, 중간 보고서, 개발 보고서가 모두 작성되었는가? |
| 8 | **QA 보고서** | 내부 QA + 교차 QA 보고서 제출 + 스크린샷 첨부 완료? |
| 9 | **BaseEntity 준수** | 모든 엔티티가 감사 필드 7종을 포함하는가? |

#### PM 반려 시 필수 제공 항목

PM이 머지 요청을 반려할 경우 **반드시 아래 3가지를 PR 코멘트에 명시**:

```markdown
### 🔴 머지 반려

**1. 반려 사유**
- (구체적인 문제점 기술)

**2. 충돌/영향 분석**
- 어떤 팀/서비스와 충돌 또는 영향이 있는지 명시
- 관련 파일 및 라인 번호 지정

**3. 수정 방향 제시**
- 어떻게 수정하면 승인 가능한지 구체적 가이드 제공
- 필요 시 참고 코드/문서 링크 첨부
```

#### ✅ 승인 케이스

TM-BE: "전표 서비스 CRUD API 구현 완료, develop 머지 요청합니다."
PM: "✅ 승인. 타 팀 충돌 없음, CI 통과, API 명세 일치 확인."

#### 🔴 반려 케이스

TM-FE: "전표 수정 이력 UI 구현 완료, develop 머지 요청합니다."
PM: "🔴 반려.
  1. 반려 사유: 전표 상태 API 응답 필드명이 백엔드 스펙과 불일치
     (FE: 'slipStatus' / BE: 'status')
  2. 충돌 분석: Slip Service API v2 응답 스키마 참조 필요
     (docs/backend/api_docs/slip_api.md#L45)
  3. 수정 방향: API 클라이언트의 필드 매핑을 BE 스펙에 맞춰 수정하거나,
     BE 팀과 협의하여 필드명 통일 후 재요청 바랍니다."
